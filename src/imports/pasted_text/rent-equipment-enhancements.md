Задача: доработать существующий экран “Техника” и карточку техники в системе управления арендой подъемников.
Важно: не менять текущий дизайн. Не делать редизайн, не менять визуальный стиль, цветовую палитру, типографику, размеры элементов, сетку, боковое меню, общую композицию страниц и текущую UX-логику. Все новые элементы должны выглядеть как органичное продолжение уже существующего интерфейса.

⸻

1. Доработать раздел “Техника”

Добавить функционал создания новой единицы техники.

При создании техники должны быть поля:
	•	производитель;
	•	модель;
	•	серийный номер;
	•	инвентарный номер;
	•	тип техники;
	•	тип привода;
	•	моточасы;
	•	год выпуска;
	•	дата ЧТО;
	•	дата ПТО;
	•	владелец техники;
	•	статус техники;
	•	локация;
	•	высота подъема;
	•	примечание.

Форма добавления техники должна быть выполнена в текущем минималистичном стиле интерфейса, без визуального усложнения.

⸻

2. Доработать карточку техники

Внутри карточки техники нужно добавить и структурировать несколько смысловых блоков.

2.1. Основные характеристики техники

В карточке техники должна быть возможность хранить и просматривать характеристики, чтобы менеджер по аренде мог открыть карточку и быстро посмотреть параметры техники.

Примеры полей:
	•	производитель;
	•	модель;
	•	серийный номер;
	•	инвентарный номер;
	•	тип;
	•	привод;
	•	год выпуска;
	•	моточасы;
	•	высота подъема;
	•	грузоподъемность;
	•	рабочая высота;
	•	габариты;
	•	масса;
	•	локация;
	•	текущий статус;
	•	текущий клиент, если техника в аренде;
	•	примечания.

⸻

2.2. История ремонтов

Добавить в карточку техники отдельный блок или вкладку “История ремонтов”.

В этом разделе должна быть возможность:
	•	вручную добавлять записи о ремонтах и обслуживании внутри приложения;
	•	в будущем автоматически получать записи из Telegram-бота или другого бота и фиксировать их в карточке техники.

Для каждой записи ремонта предусмотреть:
	•	дата;
	•	тип события (ремонт / обслуживание / диагностика / поломка);
	•	описание работ;
	•	комментарий;
	•	исполнитель / механик;
	•	статус;
	•	сумма, если требуется;
	•	источник записи: вручную или автоматически из бота.

⸻

2.3. Фото отгрузки и приемки

Добавить функционал фиксации фото отгрузок и приемки через бота с последующим хранением в приложении.

В карточке техники нужен отдельный блок или вкладка:
“Фото отгрузок и приемки”

Там должно быть:
	•	дата события;
	•	тип события: отгрузка / приемка;
	•	кто загрузил материалы;
	•	фотографии;
	•	комментарий;
	•	ссылка на связанную аренду, если есть.

Важно предусмотреть логику, что:
	•	фото могут добавляться вручную;
	•	фото могут приходить из бота автоматически;
	•	все фото должны храниться внутри карточки техники как история визуальных событий.

⸻

2.4. История аренды

В карточке техники добавить отдельную вкладку или блок “История аренды”.

Там должна отображаться история всех аренд по данной технике:
	•	клиент;
	•	даты аренды;
	•	статус аренды;
	•	менеджер;
	•	стоимость, по которой техника была сдана;
	•	длительность аренды;
	•	ссылка на карточку аренды;
	•	финансовый результат по конкретной аренде.

Список должен быть структурированным, читаемым и вписанным в текущий стиль интерфейса.

⸻

2.5. Финансовые показатели техники

В карточке техники добавить отдельный блок “Финансовые показатели”.

В нем должны быть:
	•	плановый доход техники в месяц;
	•	фактический доход за текущий месяц;
	•	утилизация за текущий месяц;
	•	количество дней в аренде за текущий месяц;
	•	количество свободных дней;
	•	дебиторка по данной технике, если применимо;
	•	финансовый результат по арендам за период.

Важно: для понимания утилизации и эффективности техника должна иметь поле:
“Плановый доход в месяц”

⸻

3. Владелец техники

В карточке техники и при создании техники добавить обязательное поле:
“Владелец техники”

Варианты владельца:
	•	собственная техника;
	•	техника инвестора;
	•	субаренда.

Это поле должно использоваться далее в отчетности и расчете финансового результата менеджера по аренде.

⸻

4. Логика расчета для отчетов менеджера

Предусмотреть, что тип владельца техники влияет на финансовую модель и отчет по менеджеру за месяц.

Если техника собственная

Менеджер по аренде получает:
	•	3% от результата по этой технике.

Если техника инвестора

Используется логика:
	•	100% выручки;
	•	минус 60%;
	•	от оставшихся 40% менеджер получает 7%.

Если техника субаренда

При выборе варианта “Субаренда” обязательно должно появляться дополнительное поле:
	•	стоимость, по которой взяли технику

Дальше система должна использовать формулу:

финансовый результат = стоимость, по которой сдали - стоимость, по которой взяли

Эту логику нужно предусмотреть в структуре интерфейса и данных, даже если полный расчет будет реализован позже.

⸻

5. UX-требования
	•	Не менять текущую навигацию.
	•	Не ломать существующую карточку техники.
	•	Новые вкладки и блоки встроить в текущую структуру нативно.
	•	Сохранить минималистичный и строгий стиль.
	•	Сделать так, чтобы карточка техники стала рабочим инструментом менеджера по аренде, сервиса и руководителя.
	•	Все новые поля и разделы должны выглядеть как часть существующего продукта, а не как отдельный новый модуль.

⸻

6. Что нельзя делать
	•	Не делать редизайн.
	•	Не менять левое меню.
	•	Не менять общую визуальную концепцию.
	•	Не усложнять интерфейс лишними декоративными элементами.
	•	Не убирать уже существующие блоки, а аккуратно расширить их.

⸻

7. Ожидаемый результат

Обновленный экран “Техника” и карточка техники, где:
	•	можно добавлять технику с нужными полями;
	•	есть характеристики техники;
	•	есть история ремонтов;
	•	есть история фото отгрузок и приемки;
	•	есть история аренды;
	•	есть финансовые показатели техники;
	•	есть указание владельца техники;
	•	предусмотрена логика для последующего расчета финансового результата менеджера.

⸻

Короткая версия промта

Не меняй текущий дизайн, стиль, сетку, шрифты, цвета и структуру интерфейса. Нужно только расширить функционал раздела “Техника” и карточки техники.

Добавь возможность создания техники с полями:
	•	производитель;
	•	модель;
	•	серийный номер;
	•	инвентарный номер;
	•	моточасы;
	•	год выпуска;
	•	дата ЧТО;
	•	дата ПТО;
	•	владелец техники;
	•	статус;
	•	локация;
	•	примечание.

В карточке техники добавь:
	•	характеристики техники;
	•	историю ремонтов с возможностью ручного заполнения и будущей автоматической фиксации из бота;
	•	историю фото отгрузок и приемки с хранением фото в приложении;
	•	историю аренды;
	•	финансовые показатели техники;
	•	плановый доход техники в месяц;
	•	поле “владелец техники”.

Варианты владельца:
	•	собственная;
	•	инвестора;
	•	субаренда.

Если техника собственная — менеджер получает 3%.
Если техника инвестора — от 40% результата менеджер получает 7%.
Если субаренда — обязательно поле “стоимость, по которой взяли”, и формула:
стоимость, по которой сдали - стоимость, по которой взяли.

Все новые элементы должны выглядеть как естественное продолжение текущего интерфейса.
